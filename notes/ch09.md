# Chapter 9: Running Interactive Scripts

## Added

- `dukpy` executes JavaScript snippets and page scripts.
- `JSContext` owns one `dukpy.JSInterpreter` per tab.
- `src/runtime.js` defines small browser-like JavaScript objects.
- `console.log` calls back into Python's `print`.
- External `<script src="...">` files are loaded and run during `Tab.load`.
- `document.querySelectorAll` calls into Python selector matching.
- Python DOM nodes are represented in JavaScript by integer handles.
- JavaScript wraps handles in lightweight `Node` objects.
- `Node.getAttribute` passes the handle back to Python to read attributes.
- `Node.addEventListener` stores JavaScript callbacks by handle and event type.
- Python dispatches click, keydown, and submit events into the JS runtime.
- Event listeners run with `this` set to the event target `Node`.
- `Event.preventDefault` lets JavaScript cancel browser default actions.
- `Node.innerHTML` calls back into Python to replace an element's children.
- DOM changes call `Tab.render()` so layout and paint data are regenerated.
- The guest book now uses JavaScript to warn about long comments and cancel
  invalid submissions.
- The server still rejects long comments so validation does not depend on JS.

## Key Ideas

- DukPy is only a JavaScript engine; browser APIs like `document` must be added.
- `export_function` exposes selected Python functions to JavaScript.
- Handles avoid trying to pass Python `Element` objects directly into JavaScript.
- One `JSContext` per tab lets multiple scripts on the same page share variables.
- Event listeners stay in JavaScript because they are JavaScript functions.
- Event dispatch crosses from Python to JavaScript, while DOM methods still call
  back from JavaScript to Python through handles.
- `innerHTML` parses a fragment by wrapping it in temporary `html` and `body`
  elements, then moving the parsed body children onto the target element.
- Client-side validation improves interaction, but server-side validation still
  owns correctness.

## Bugs / Fixes

- Local files should be tested through a local HTTP server, not `./index.html`.
- Page scripts should run through `JSContext`, not bare `dukpy.evaljs`, so the
  runtime objects are available.
- `dispatch_event` belongs on `JSContext`, not `Tab`, because it needs the
  JavaScript interpreter and handle maps.
- `innerHTML_set` must be exported to DukPy; otherwise the JavaScript runtime's
  `call_python("innerHTML_set", ...)` cannot reach Python.
- Server assets should be opened relative to `src/server.py`, not the shell's
  current working directory.

## Python Notes

- Dictionaries are used in both directions: node to handle, and handle to node.
- `os.path.dirname(__file__)` makes `runtime.js` load relative to `browser.py`.
