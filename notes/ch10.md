# Chapter 10: Keeping Data Private

## Added

- Browser cookie storage with `COOKIE_JAR`.
- `Cookie` request headers for hosts with stored cookies.
- `Set-Cookie` response handling.
- Server-side session tokens and `SESSIONS`.
- Guest book login form and fixed demo accounts.
- Logged-in user tracking in the server-side session.
- Logged-in-only guest book posting.
- Author names on guest book entries.
- Minimal synchronous `XMLHttpRequest`.
- Same-origin checks for XHR.
- CSRF nonce generation and validation for the posting form.

## Key Ideas

- Cookies give the browser a server-granted identity.
- The cookie should identify a server-side session, not store all user data.
- XHR includes cookies, so cross-origin XHR could leak private data.
- Same-origin policy compares scheme, host, and port.
- CSRF does not need to read private data; it makes the user's browser act with
  the user's cookies.
- A nonce gives each legitimate form a server-generated secret value.

## Bugs / Fixes

- New visitors must receive a `Set-Cookie` header; otherwise login state is not
  preserved across requests.
- Posting after `add_entry` should re-render with the session, not submitted
  params.
- `ENTRIES.append` needs a single tuple argument containing entry and author.
- XHR must be exported to DukPy before JavaScript can call back into Python.
- XHR should check same-origin before sending the request.

## Python Notes

- `dict.setdefault(key, default)` gets an existing session or creates a new one.
- Tuples group each guest book entry with its author.
- Returning early is a simple way to reject unauthorized or invalid requests.
